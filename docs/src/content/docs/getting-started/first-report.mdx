In the next few minutes you will scaffold a report bundle, preview it live in your browser, and export a PDF — the core loop you will use every day with bino.

This guide assumes you have `bino` installed and available on your `PATH`.

<Steps>

1.  **Create a new workdir**

    A **workdir** is the root directory of a report bundle.
    Run `bino init` to scaffold a new project:

    ```bash
    bino init
    ```

    `bino init` asks a few questions — target directory, report name, title, language — then generates the starter files. Use `-y` / `--yes` to accept defaults non-interactively.

    <img src={initImg.src} alt="Terminal output of bino init showing the interactive prompts and generated files" style="max-width:100%;border-radius:0.5rem;" />

    <Aside>
      You can specify a custom directory name using the `--directory` flag:
      ```bash bino init --directory my-report ```
    </Aside>

2.  **Inspect the generated files**

    Navigate into the newly created directory:

    ```bash
    cd rainbow-report
    ```

    The scaffolded workdir contains everything bino needs:

    | File | Purpose |
    |------|---------|
    | `bino.toml` | Project root marker and configuration |
    | `data.yaml` | **DataSource** (loads raw data) + **DataSet** (SQL query) |
    | `pages.yaml` | **LayoutPage** — page layout with child components |
    | `report.yaml` | **ReportArtefact** — assembles pages into a PDF |

    Each manifest shares the same envelope: `apiVersion: bino.bi/v1alpha1` and a `kind` that tells bino what role the document plays. See [Key ideas](/getting-started/key-ideas/) for the full mental model.

3.  **Preview the report**

    Start a live preview server from the workdir:

    ```bash
    bino preview
    ```

    bino scans the directory for manifests, starts a local HTTP server (by default `http://127.0.0.1:45678/`), and opens your default browser.
    As you edit YAML or data files, it reloads the preview automatically.

    <Aside type="tip">
      Run `bino preview --log-sql` to see the SQL queries used to populate
      datasets.
    </Aside>

4.  **Build a PDF**

    When you are happy with the preview, export a PDF:

    ```bash
    bino build
    ```

    bino validates manifests, executes datasets in the embedded query engine, renders HTML, and converts pages to PDF. Output goes to `dist/` by default.

    Open `dist/rainbow-sample-report.pdf` — you should see the chart rendered with your data:

    <img src={chartImg.src} alt="The revenue-by-region bar chart generated by bino build" style="max-width:360px;border:1px solid var(--sl-color-gray-5);border-radius:0.5rem;" />

    Example using a custom output directory:

    ```bash
    bino build --out-dir dist/reports
    ```

5.  **Export component screenshots** *(optional)*

    Besides PDFs, bino can capture individual components as PNG or JPEG images using `ScreenshotArtefact`. This is useful for embedding charts in presentations, dashboards, or documentation.

    Add a `screenshots.yaml` file to your workdir:

    ```yaml
    apiVersion: bino.bi/v1alpha1
    kind: ScreenshotArtefact
    metadata:
      name: chart_export
    spec:
      filenamePrefix: export
      format: xga
      scale: device
      refs:
        - kind: ChartStructure
          name: revenue_chart
    ```

    <Aside type="caution">
      The component must have a `metadata.name` on the inline child in your `LayoutPage` for the screenshot to target it. For example:
      ```yaml
      children:
        - kind: ChartStructure
          metadata:
            name: revenue_chart  # targetable by ScreenshotArtefact
          spec:
            dataset: revenue_by_region
      ```
    </Aside>

    Run `bino build` again. Alongside the PDF, you will find `export-revenue_chart.png` in the output directory — a crisp, retina-resolution image of just that chart.

    See [ScreenshotArtefact reference](/reference/screenshot-artefact/) for all options including JPEG output, indexed filenames, and transparent backgrounds.

</Steps>

## What you just built

You created a complete report bundle: a data source feeds into a SQL query, the query powers a chart, and a layout page composes it into a PDF. This is the [Reporting as Code](/why/reporting-as-code/) workflow — everything is a text file, everything is reproducible, and the same `bino build` command works on your laptop and in CI.

The images on this page were themselves generated by bino's `ScreenshotArtefact` — the same tool you just learned about.
